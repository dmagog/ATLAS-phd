### 2. Дописать ещё 50 QA-entries
Распределение по разделам программы кандэкзамена (требует уточнения после M4.5):
- Геометрическая оптика — ~10
- Волновая оптика (интерференция, дифракция, поляризация) — ~20
- Распространение в средах, дисперсия — ~10
- Лазерная оптика — ~5
- Прикладная оптика (приборы, эксперименты) — ~5

Подход (см. roadmap M3.A):
1. Hand-craft базовых 30 как калибровка (10 уже есть → +20 руками).
2. LLM-assisted draft для оставшихся 30 (генерируем кандидатов из chunks корпуса) → ручная валидация.

### 4. Refusal-блок до 20
Текущие 3 — слишком очевидные (бытовое, историческое, далёкое физическое). Нужно ещё 17, в том числе **adversarial**:
- Близкие, но out-of-corpus темы (например, нелинейная оптика — если корпус её не содержит)
- Вопросы с общей частью (например, «расскажи о телескопе Hubble» — оптика есть, но фактология астрономическая)
- Запросы с подвохом для prompt injection
- Запросы, требующие ground-truth, которого в корпусе быть не должно (например, «какие были оценки экзаменаторов в 1985?»)

**Все refusal-блоки — вручную, без LLM (иначе систематический bias).**

### 5. Self-check блок (20 entries)
Каждая запись — пара (canned_question, user_answer) с эталонной оценкой по рубрике (correctness/completeness/logic/terminology, шкала 0..5).

Подход:
1. Hand-craft 10 типичных multiple-choice + open-ended вопросов по основным темам.
2. Для каждого — несколько user_answer вариантов с разной правильностью (полный, частичный, неверный).
3. Эталонная оценка от экспертов (≥ 2 человека, медиана).

## Schema

См. `eval/schema.py`. Каждая JSONL-запись валидируется через `parse_entry()` или `load_jsonl()`.

Pydantic-validation вызывается:
```python
from eval.schema import load_jsonl
from pathlib import Path
entries = load_jsonl(Path("eval/golden_set_v1/golden_set_v1.0.jsonl"))
```

При schema mismatch — `ValueError` с указанием номера строки.

## Versioning

- `v1.0-draft` — текущее состояние, pages не заполнены, набор не полный.
- `v1.0` — frozen после прохождения checklist выше; от этого момента файл immutable, изменения = новая версия.
- `v1.x` — пополнения внутри той же мажор-версии (без изменения schema).
- `v2.x` — после M4.5 (расширение под полную программу кафедры).
