# Use Cases и критерии приемки v1.1 (MVP)

## UC-01: Ответ на вопрос по теме (Q&A)

### Цель
Получить обоснованный ответ по учебному вопросу с прозрачными источниками.

### Акторы
- Пользователь.
- Система ATLAS.

### Предусловия
- Пользователь аутентифицирован.
- База знаний загружена и проиндексирована.
- Активирован режим `Q&A`.

### Основной поток
1. Пользователь отправляет вопрос.
2. `Planner` определяет маршрут запроса (`qa/self-check/clarify`).
3. Система выполняет retrieval и rerank релевантных фрагментов.
4. `Verifier` проверяет guardrails (порог релевантности и достаточность evidence).
5. При прохождении порога система генерирует ответ с источниками.
6. Система сохраняет Q&A в контекст текущей сессии и логирует метаданные решения.

### Альтернативный поток A: недостаточная релевантность
1. `Verifier` не пропускает ответ (hard-gate).
2. Система выдает отказ в уверенном ответе.
3. Система объясняет причину и предлагает 1-3 варианта переформулировки.
4. Событие отказа фиксируется в логах с `reason_code`.

### Критерии приемки
- Невозможно получить содержательный ответ без источников.
- При недостатке контекста система отказывает, а не генерирует недостоверный ответ.
- Решение `Planner` и исход `Verifier` логируются.
- Q&A контекст доступен в пределах текущей сессии.

### Связанные тесты
- AT-01, AT-07, AT-08, AT-09, AT-14.

## UC-02: Самопроверка по теме (Self-check)

### Цель
Оценить уровень понимания темы по смешанному набору вопросов.

### Акторы
- Пользователь.
- Система ATLAS.

### Предусловия
- Пользователь аутентифицирован.
- Активирован режим `Self-check`.

### Основной поток
1. Пользователь указывает тему самопроверки.
2. `Planner` маршрутизирует запрос в ветку `Self-check`.
3. Система формирует набор вопросов:
   - тестовые;
   - открытые.
4. Пользователь отправляет ответы.
5. `Evaluator` оценивает:
   - тестовые по ключам;
   - открытые по рубрике (критерии + веса).
6. `Feedback Agent` формирует итоговую обратную связь и система возвращает:
   - итоговый балл;
   - покритериальную оценку;
   - рекомендации по улучшению.
7. Результат попытки сохраняется в истории между сессиями.

### Альтернативный поток A: невалидная оценка
1. Payload `Evaluator` не проходит валидацию схемы.
2. Попытка маркируется как `invalid_evaluation`.
3. Пользователю возвращается контролируемое сообщение о невозможности завершить оценку.
4. Событие фиксируется в логах качества.

### Критерии приемки
- В каждой попытке есть оба типа вопросов.
- Для открытых ответов выдается не только балл, но и объяснение по критериям.
- Результаты попыток доступны пользователю после завершения.
- Невалидная оценка не публикуется как финальный результат.

### Связанные тесты
- AT-03, AT-10, AT-11, AT-12, AT-14.

## UC-03: Пакетная загрузка материалов администратором

### Цель
Обновить базу знаний через контролируемый импорт.

### Акторы
- Администратор.
- Система ATLAS.

### Предусловия
- Пользователь аутентифицирован как `администратор`.

### Основной поток
1. Администратор загружает пакет файлов.
2. Система валидирует формат каждого файла.
3. Система создает `ingestion_job` и запускает обработку:
   - extraction;
   - normalization;
   - chunking;
   - embeddings;
   - indexing.
4. Система показывает статусы по каждому файлу и итоговый статус job.

### Альтернативный поток A: неподдерживаемый формат
1. Администратор загружает `djvu`.
2. Система отклоняет файл, возвращает понятную ошибку формата.

### Альтернативный поток B: попытка неадминистратора
1. Пользователь пытается вызвать загрузку.
2. Система блокирует действие и возвращает `403/forbidden`.
3. Событие безопасности логируется.

### Критерии приемки
- Только администратор имеет доступ к загрузке и индексации.
- Поддерживаемые форматы принимаются, неподдерживаемые отклоняются явно.
- Повторная загрузка файла не дублирует chunk при совпадении SHA-256.

### Связанные тесты
- AT-04, AT-05, AT-06.

## UC-04: Работа с формулами

### Цель
Сохранить корректность математической и технической нотации.

### Акторы
- Пользователь.
- Система ATLAS.

### Предусловия
- Вопрос или ответ содержит формулу.

### Основной поток
1. Система формирует формулу в LaTeX-представлении.
2. В web UI формула рендерится визуально.
3. В Telegram формула отправляется LaTeX-текстом (MVP-режим).

### Критерии приемки
- Формулы не исчезают из ответа.
- В web UI формулы читаемы и рендерятся корректно.

### Связанные тесты
- AT-02, AT-13.

## UC-05: Двуязычное взаимодействие

### Цель
Поддерживать работу системы на русском и английском языках.

### Акторы
- Пользователь.
- Система ATLAS.

### Предусловия
- Пользователь отправляет запрос на русском или английском.

### Основной поток
1. Система определяет язык запроса или использует явно указанный язык.
2. Система формирует ответ на том же языке.
3. Формат источников сохраняется одинаковым для обоих языков.

### Критерии приемки
- UC-01 и UC-02 проходят на русском и английском.

### Связанные тесты
- AT-12.
