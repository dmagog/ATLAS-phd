# ATLAS phd: Product Proposal (Milestone 1)

## 1. Обоснование идеи

ATLAS phd решает прикладную задачу подготовки к кандидатскому минимуму в технической области.
Пользователю нужно быстро получать объяснения по теме, проверять понимание и опираться на проверяемые источники, а не на "средний" чат-ответ без контроля достоверности.

Почему это важно:
- материалы разрознены и объемны;
- высокая цена ошибки: недостоверный ответ ухудшает подготовку;
- нужен короткий цикл обучения с обратной связью по пробелам.

Подход проекта: агентный контур над RAG, где отдельные роли (планирование, извлечение, генерация, верификация, самопроверка) повышают управляемость и надежность.

Почему выбран именно `agentic RAG`, а не альтернатива:
для предметной области с формулами, источниками и честными отказами plain RAG недостаточно управляем, потому что в нем хуже разделены маршрутизация, генерация и проверка результата. При этом полностью "свободный" tool-calling агент для PoC избыточен и сложнее верифицируется. Явный граф поверх RAG дает компромисс: сохраняет простую и проверяемую retrieval-базу, но добавляет управляемые agentic-узлы там, где они реально повышают надежность.

## 2. Цель и метрики

### 2.1 Цель проекта

Собрать PoC агентного учебного ассистента, который:
- отвечает на вопросы по базе материалов с источниками;
- честно отказывает при недостатке данных;
- проводит self-check и дает полезную обратную связь;
- работает через web-чат как основной интерфейс и через Telegram как дополнительный канал.

### 2.2 Метрики успеха

#### Продуктовые метрики

| Метрика | Цель PoC | Как измеряем |
|---|---|---|
| Task completion rate (Q&A + self-check) | >= 80% | Доля сценариев демо-набора, завершенных без ручной помощи |
| Полезность обратной связи self-check | >= 4.0/5 | Экспертная оценка пилотных ответов |
| Доля ответов с источниками | >= 95% | Логи ответов в Q&A |

#### Агентские метрики

| Метрика | Цель PoC | Как измеряем |
|---|---|---|
| Точность роутинга Planner Agent | >= 90% | Сравнение маршрута с ручной разметкой набора запросов |
| Доля корректных отказов при низкой релевантности | >= 85% | Контрольный off-topic/low-context набор |
| Доля ответов, прошедших Verifier без критических замечаний | >= 90% | Логи верификатора |

#### Технические метрики

| Метрика | Цель PoC | Как измеряем |
|---|---|---|
| p95 latency Q&A | <= 15 сек | Замеры API/бота |
| Error rate (5xx/timeout) | <= 3% | Технические логи |
| Успешность ingestion-job | >= 95% | Статусы пакетного импорта |

## 3. Сценарии использования и edge-кейсы

### 3.1 Основные сценарии

1. Пользователь задает вопрос по теме, получает ответ с источниками.
2. Пользователь запускает self-check по теме, получает оценку и рекомендации.
3. Администратор загружает пакет материалов и обновляет базу знаний.

### 3.2 Edge-кейсы

1. Недостаточная релевантность контекста: система не отвечает "по памяти", а выдает отказ и варианты переформулировки.
2. Конфликтные или устаревшие фрагменты в KB: система указывает источники и сообщает о неопределенности.
3. Попытка prompt injection в пользовательском запросе или документе: системные инструкции и политика безопасности имеют приоритет.
4. Неподдерживаемый формат файла (`djvu`) при загрузке: явное отклонение с понятной причиной.
5. Перегрузка или деградация внешнего LLM API (`429/5xx`): retry и controlled error.

## 4. Ограничения

### 4.1 Технические ограничения (предварительно)

- Локальное развертывание MVP, без production-кластера.
- Основной демонстрационный и пользовательский поток MVP проходит через web-интерфейс, поскольку предметная область требует удобного рендера формул и структурированного вывода.
- `p95 latency` для Q&A: целевое значение до 15 секунд.
- Ограниченный набор форматов в ingestion: `pdf`, `docx`, `txt`, `md`.
- Ограниченная область знаний: один профильный экзамен.

### 4.2 Операционные ограничения (предварительно)

- Команда и сроки: короткий цикл разработки, фокус на PoC.
- Бюджет внешнего LLM API: целевой лимит до 150 USD/месяц на этапе пилота.
- Источники данных загружаются вручную администратором, без автоматического web-crawling.

### 4.3 Грубый расчет бюджета API

Плановый расчет для PoC:
- 8-10 активных пользователей в пилоте;
- 8-10 взаимодействий в день на пользователя;
- около 2,500-3,000 LLM-вызовов в месяц;
- средний объем: 4,000-6,000 суммарных токенов на один `Q&A` запрос и 6,000-9,000 токенов на один `self-check` сценарий;
- при planning assumption порядка 8-10 USD за 1M blended tokens месячный бюджет попадает примерно в диапазон 90-150 USD.

Это не финальная смета, а грубая budgeting-модель PoC; точные расходы будут зависеть от выбранной модели, размера контекста и доли `self-check` сценариев.

## 5. Архитектурный набросок

Примечание:
- В разделе 5-6 зафиксирована целевая архитектура.
- Для PoC текущего этапа применяются ограничения, перечисленные в разделе 7.1 (в частности, упрощенная ветка `self-check`).

### 5.1 Модули

- `Planner Agent`: классификация типа запроса и выбор ветки выполнения.
- `Retrieval Tool`: поиск релевантных chunk в индексе (`PostgreSQL + pgvector`).
- `Answer Agent`: формирование ответа по найденным источникам.
- `Verifier Agent`: проверка evidence coverage, применение guardrails, решение `answer/refuse`.
- `Self-check Question Generator`: генерация набора вопросов по теме.
- `Self-check Evaluator`: оценка ответов по рубрике и критериям.
- `Self-check Feedback Agent`: формирование развивающей обратной связи и рекомендаций.
- `API/UI слой`: `FastAPI`, web-чат как основной интерфейс, Telegram-бот как дополнительный канал.
- `Storage`: пользователи, истории, результаты self-check, ingestion-метаданные, логи.

### 5.2 Внешние интеграции

- LLM API для reasoning/generation.
- Telegram Bot API.
- Локальная БД PostgreSQL (включая pgvector).

## 6. Потенциальный data flow и делегирование в LLM/Agent

Ниже приведен целевой data flow; фактическая PoC-конфигурация указана в разделе 7.1.

1. Пользовательский запрос поступает в API.
2. `Planner Agent` определяет маршрут (`Q&A` или `self-check`).
3. Для `Q&A`: retrieval -> answer generation -> verifier -> ответ или отказ.
4. Для `self-check`: `Question Generator` -> `Evaluator` -> `Feedback Agent`.
5. Результаты и метаданные сохраняются в storage.

Что делегируем LLM/Agent:
- классификацию намерения;
- формирование ответа/объяснения;
- генерацию вопросов self-check;
- оценку ответов по рубрике;
- генерацию обратной связи по итогам оценки.

Что не делегируем LLM:
- аутентификацию/авторизацию (RBAC);
- применение hard guardrails и порогов релевантности;
- принятие административных действий (загрузка/удаление данных без явного запроса и прав);
- политику хранения данных и аудит-логирование.

### 6.1 Контракт `Self-check Evaluator -> Feedback Agent`

Зафиксирован формат `C (hybrid)`: структурированные поля + краткое текстовое объяснение.

Обязательные поля:
- `attempt_id`, `user_id`, `topic`, `language`, `timestamp`;
- `overall_score` (шкала `0..5`);
- `criterion_scores` (оценки `0..5` по критериям рубрики);
- `question_results` (`question_id`, `question_type`, `score`, `status`);
- `error_tags` (например: `terminology`, `incomplete`, `logic_gap`);
- `confidence` (`0..1`);
- `evaluator_summary` (2-4 предложения);
- `policy_flags` (`low_confidence`, `inconsistent_eval`, `needs_review`).

Цель контракта:
- обеспечить воспроизводимость оценки;
- упростить расчет метрик и последующий A/B анализ;
- стабилизировать качество рекомендаций `Feedback Agent`.

### 6.2 State-модель графа (статусы и переходы)

#### Основные статусы

| Статус | Назначение | Следующие переходы |
|---|---|---|
| `REQUEST_RECEIVED` | Запрос принят API | `PLANNER_DECIDED`, `TECHNICAL_ERROR` |
| `PLANNER_DECIDED` | Определен маршрут (`qa`, `self_check`, `clarify`) | `QA_RETRIEVAL_DONE`, `SC_QUESTIONS_READY`, `CLARIFICATION_REQUESTED` |
| `QA_RETRIEVAL_DONE` | Получены кандидаты retrieval | `QA_ANSWER_DRAFTED`, `REFUSAL_SENT`, `TECHNICAL_ERROR` |
| `QA_ANSWER_DRAFTED` | Сформирован черновик ответа с источниками | `QA_VERIFIED_PASS`, `QA_VERIFIED_FAIL` |
| `QA_VERIFIED_PASS` | Проверка пройдена | `RESPONSE_SENT` |
| `QA_VERIFIED_FAIL` | Проверка не пройдена (hard-gate) | `REFUSAL_SENT` |
| `SC_QUESTIONS_READY` | Вопросы self-check подготовлены | `SC_EVALUATED`, `TECHNICAL_ERROR` |
| `SC_EVALUATED` | Ответы оценены по рубрике | `SC_FEEDBACK_READY`, `INVALID_EVALUATION` |
| `SC_FEEDBACK_READY` | Подготовлен финальный feedback | `RESPONSE_SENT` |
| `INVALID_EVALUATION` | Нарушен контракт оценки | `REFUSAL_SENT`, `TECHNICAL_ERROR` |

#### Терминальные исходы

- `RESPONSE_SENT`: корректный ответ или результат self-check отправлен пользователю.
- `REFUSAL_SENT`: осознанный отказ с объяснением и следующим шагом (уточнение/переформулировка).
- `CLARIFICATION_REQUESTED`: запрос уточнения до повторного запуска графа.
- `TECHNICAL_ERROR`: контролируемая техническая ошибка с `request_id`.

#### Коды причин отказа (терминальные для `REFUSAL_SENT`)

- `low_relevance_top1` — top-1 ниже порога релевантности.
- `insufficient_evidence` — недостаточно независимых релевантных фрагментов.
- `conflicting_evidence` — источники противоречат друг другу.
- `missing_citations` — черновик ответа не содержит валидных ссылок на источник.
- `policy_blocked` — ответ заблокирован политикой безопасности.
- `invalid_evaluation` — self-check оценка не прошла проверку контракта.

### 6.3 Eval-set для пилотной проверки

Пилотный `eval-set v0` планируется как набор из 120 элементов:
- 60 запросов `Q&A` по реальным учебным материалам корпуса;
- 20 off-topic или low-context запросов для проверки честных отказов;
- 20 вопросов с формулами или терминологически плотными фрагментами;
- 20 примеров `self-check` с эталонной экспертной оценкой открытых ответов.

Источник данных:
- загруженные учебные материалы корпуса;
- вопросы из подготовленных тем курса;
- пилотный набор hand-crafted adversarial и refusal cases.

Процесс разметки:
- первичная разметка выполняется участником проекта по шаблону ответа/отказа;
- спорные и пограничные кейсы проходят вторичную проверку;
- для `self-check` эталонная оценка фиксируется по рубрике проекта (`0..5` и покритериальные баллы);
- после каждой итерации PoC набор не переписывается целиком, а версионируется и дополняется.

## 7. Зона роста проекта (после PoC)

### 7.1 Зафиксированные решения для текущего PoC

- Оркестрация: явный граф агентов/узлов (`Planner -> Retrieval -> Answer -> Verifier`).
- Ветка `self-check` в PoC: 2 шага (`Question Generator -> Evaluator`) с базовой обратной связью.
- Политика надежности: `hard-gate` верификатор (при недостатке evidence ответ блокируется).
- Retrieval в PoC: только `vector search` как базовый и управляемый вариант.

### 7.2 Планируемый апгрейд retrieval

- Следующий шаг: переход к `hybrid retrieval` (`BM25 + vector`) без смены общей агентной архитектуры.
- Триггер апгрейда: рост доли отказов/промахов на терминологических, формульных и keyword-heavy запросах.
- Ожидаемый эффект: повышение recall и снижение ложных отказов при сохранении `hard-gate` политики.

### 7.3 Техническая подготовка к апгрейду

- Сохранять retrieval за единым интерфейсом (`retrieve(query) -> candidates`), чтобы заменить стратегию без рефакторинга графа.
- Логировать причины отказа (`low_relevance`, `insufficient_evidence`) и тип запроса для анализа качества retrieval.
- Вести A/B сравнение `vector-only` vs `hybrid` на контрольном наборе до переключения по умолчанию.

### 7.4 Память и история обучения (текущее решение PoC)

- Принят компромисс `A+`:
  - `session-only memory` для текущего диалога;
  - сохранение базовой истории `self-check` между сессиями (попытка, тема, балл, критерии, краткий feedback).
- Цель: сохранить полезную историю попыток без внедрения сложной персонализации в PoC.

### 7.5 Зона роста по персонализации (что НЕ делаем в PoC)

- Не строим долговременный профиль слабых/сильных тем с авто-адаптацией.
- Не меняем стратегию генерации вопросов/ответов на основе полной истории пользователя.
- Не строим персональные learning-path и продвинутые дашборды прогресса.
- Не подмешиваем автоматически старые попытки в каждый новый ответ `Q&A`.

### 7.6 План расширения self-check до полной 3-шаговой схемы

- Целевая архитектура: `Question Generator -> Evaluator -> Feedback Agent`.
- Этап 1 (PoC): 2 шага (`Question Generator + Evaluator`), базовая обратная связь формируется из результата оценки.
- Этап 2 (расширение): выделение отдельного `Feedback Agent` с более качественной педагогической обратной связью.
- Критерий перехода к этапу 2: стабильная базовая точность оценки и наличие запроса на более глубокие рекомендации в UX.
