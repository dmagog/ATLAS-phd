# Техническое задание (ТЗ) v1.1
## Проект ATLAS phd (Assistant for Technical Learning & Attestation Support)

## 1. Общие сведения

- Наименование системы: **ATLAS**.
- Назначение: интеллектуальный ассистент для подготовки к кандидатскому минимуму по профильному техническому экзамену.
- Тип решения: агентная RAG-система с LLM через API (явный граф оркестрации).
- Каналы MVP:
  - веб-чат как приоритетный интерфейс MVP;
  - Telegram-бот как дополнительный канал.
- Языки MVP: русский и английский.

## 2. Область применения и границы MVP

### 2.1. Область применения
- Пилот охватывает один выбранный профильный технический экзамен.
- Внутри пилота допускаются материалы из нескольких курсов, относящихся к этому экзамену.

### 2.2. Что входит в MVP
- Режим `Q&A` по базе знаний.
- Режим `Self-check` (тестовые и открытые вопросы).
- Явная агентная оркестрация:
  - `Planner`;
  - `Retrieval`;
  - `Answer`;
  - `Verifier` (hard-gate);
  - `Self-check` цепочка.
- Роли `администратор` и `пользователь`.
- Ручной пакетный импорт материалов администратором.
- Локальное развертывание.
- Метрики качества RAG и автопроверки.

### 2.3. Что не входит в MVP
- Режим преподавателя.
- Нативная поддержка `djvu`.
- Production-SLA и высоконагруженная эксплуатация.
- Продвинутая долгосрочная аналитика учебного прогресса.

## 3. Роли, аутентификация и авторизация (RBAC)

### 3.1. Роли
- `Администратор`:
  - загрузка и удаление материалов;
  - запуск и мониторинг ingestion-job;
  - просмотр системных журналов операций загрузки.
- `Пользователь`:
  - вопросы и ответы;
  - самопроверка;
  - просмотр своих результатов.

### 3.2. Аутентификация по каналам
- Telegram:
  - идентификатор: `telegram_user_id`;
  - доступ администратора: allowlist по `telegram_user_id` в конфигурации.
- Web:
  - логин по email/паролю для MVP;
  - пароль хранится как `argon2/bcrypt` hash;
  - сессия через JWT (access token + срок жизни).

### 3.3. Авторизация
- Все административные endpoint и команды защищены проверкой роли.
- Проверка роли выполняется на уровне API middleware и повторно на уровне handler/service.
- Любая попытка доступа без прав логируется как событие безопасности.

## 4. Функциональные требования

**FR-01. Каналы взаимодействия**
- Система должна поддерживать веб-чат.
- Система должна поддерживать Telegram-бота.

**FR-02. Языки**
- Система должна принимать запросы и формировать ответы на русском и английском языках.

**FR-03. Режимы работы**
- Должны быть доступны режимы `Q&A` и `Self-check`.
- Должны быть доступны профили ответа: `кратко`, `подробно`, `учебный`.

**FR-04. Роли и доступ**
- Изменение базы знаний разрешено только роли `администратор`.
- Роль `пользователь` не имеет прав изменения KB.

**FR-05. Поддерживаемые форматы**
- Форматы MVP: `pdf`, `docx`, `txt`, `md`.
- Формат `djvu` в MVP не поддерживается.

**FR-06. Ручной пакетный импорт**
- Администратор загружает пакет файлов.
- Система запускает ingestion-пайплайн:
  - извлечение текста;
  - нормализация;
  - чанкинг;
  - эмбеддинги;
  - индексация.
- Для каждого файла фиксируется статус `accepted/rejected/processed/failed`.

**FR-07. Ответы с опорой на контекст**
- Ответ в `Q&A` разрешен только при прохождении порога релевантности (см. раздел 8).
- Ответ должен содержать источники.

**FR-08. Прозрачность источников**
- Каждый ответ должен включать:
  - `document_title` (или имя файла);
  - `section/page` (если доступно);
  - цитируемый фрагмент (или его однозначный указатель).

**FR-09. Честный отказ**
- При недостатке релевантных данных система обязана:
  - отказать в уверенном ответе;
  - объяснить причину;
  - предложить 1-3 варианта переформулировки.

**FR-10. Self-check**
- По теме генерируется набор из двух типов вопросов:
  - тестовые;
  - открытые.
- Ограничение по времени в MVP не применяется.

**FR-11. Автопроверка открытых ответов**
- Открытые ответы оцениваются по рубрике (см. раздел 9).
- Пользователь получает:
  - итоговый балл;
  - пояснение по критериям;
  - рекомендации по улучшению.

**FR-12. Формулы**
- Формулы передаются в LaTeX-представлении.
- В веб-чате формулы должны рендериться.
- В Telegram допускается LaTeX-текст без рендера.

**FR-13. Хранение учебных и системных данных (MVP)**
- В MVP хранятся:
  - результаты `self-check` попыток;
  - журналы админ-операций;
  - метаданные решений оркестрации для аудита качества.
- Долгосрочная персонализация и progress dashboard вынесены за рамки MVP.

**FR-14. Агентная маршрутизация и верификация**
- Перед выполнением сценария запрос проходит через `Planner`.
- В `Q&A` обязателен этап `Verifier` в режиме hard-gate:
  - при недостатке evidence ответ блокируется;
  - возвращается отказ с объяснением и вариантами переформулировки.

**FR-15. Логирование оркестрации**
- Для каждого запроса фиксируются переходы графа (`state_from/state_to`) и решение маршрутизации.
- Для отказов и блокировок обязателен `reason_code`.

**FR-16. Контракт оценки self-check**
- `Evaluator` возвращает структурированный payload оценки.
- Невалидный payload маркируется как `invalid_evaluation` и не публикуется как финальный результат.

**FR-17. Модель памяти в MVP**
- `Q&A` использует сессионную память текущего диалога.
- История `self-check` попыток хранится между сессиями пользователя.

## 5. Нефункциональные требования

**NFR-01. Стек**
- Python.
- `aiogram` + `FastAPI` + `PostgreSQL`.
- Векторный поиск: `pgvector` (см. ADR-001).

**NFR-02. Архитектура**
- Модульная структура:
  - `ui_telegram`;
  - `ui_web`;
  - `api`;
  - `orchestration`;
  - `planner`;
  - `ingestion`;
  - `retrieval`;
  - `answer`;
  - `verifier`;
  - `self_check`;
  - `storage`.

**NFR-03. Локальное развертывание**
- MVP должен запускаться локально по инструкции.
- Должен быть `docker-compose` профиль для быстрого старта.

**NFR-04. Документация**
- README с шагами запуска и конфигурации.
- Техническая документация по API и ingestion-процессу.

**NFR-05. Безопасность**
- Секреты только в переменных окружения.
- API-ключи и токены не попадают в git.
- Логируются входы администратора и изменения KB.

**NFR-06. Наблюдаемость и устойчивость**
- Структурированные логи (JSON) с `request_id`.
- Метрики: latency, error-rate, ingest-job status, refusal-rate.
- Retry для LLM-запросов при `429/5xx`: до 2 повторов с backoff.
- Таймауты:
  - retrieval: до 2 сек;
  - LLM generation: до 25 сек.

## 6. Архитектурное решение по векторному поиску (ADR-001)

- Решение MVP: хранить эмбеддинги в `PostgreSQL + pgvector`.
- Причины:
  - минимальная операционная сложность для локального пилота;
  - единая точка данных;
  - достаточно для масштаба 1-2 активных пользователей.
- Условие пересмотра:
  - при росте объема KB или требований к latency возможен переход на отдельный vector DB.

## 7. Политика данных и хранения

### 7.1. Что храним в MVP
- Пользователи и роли.
- Метаданные документов и chunk.
- Сессионный журнал Q&A текущего диалога (вопрос, ответ, источники, статус ответа/отказа).
- Попытки self-check и результаты оценивания.
- Журналы админ-действий.

### 7.2. Ретеншн и приватность
- Сессионные записи Q&A живут в пределах активной пользовательской сессии.
- Учебные записи self-check хранятся до ручной очистки администратором или запроса удаления.
- Системные логи: 90 дней (значение по умолчанию MVP).
- Пользователь может запросить удаление своей учебной истории.

### 7.3. Идемпотентность ingestion
- Уникальность файла определяется по SHA-256.
- Повторная загрузка того же файла не должна создавать дубликаты chunk.
- Каждая пакетная загрузка имеет `job_id` и финальный статус.

## 8. Политика retrieval и отказа (RAG guardrails)

### 8.1. Модель решения
- На этапе rerank рассчитывается нормализованный `relevance_score` в диапазоне `[0..1]`.
- Для ответа должны одновременно выполняться условия:
  - `top1_score >= 0.70`;
  - минимум 2 фрагмента с `score >= 0.60`.

### 8.2. Поведение системы
- Если условия выполнены: выдается ответ с источниками.
- Если не выполнены: выдается отказ по FR-09.
- Генерация ответа без источников запрещена.
- В режиме отказа система предлагает переформулировки:
  - уточнить термин;
  - сузить тему;
  - указать раздел/главу.

### 8.3. Логирование
- Для каждого ответа логируются:
  - `top_k_scores`;
  - итоговое решение `answer/refuse`;
  - причина решения.

## 9. Рубрика автопроверки открытых ответов

Итоговая шкала: `0..5`.

Критерии (веса):
- Фактическая корректность: 40%.
- Полнота покрытия вопроса: 30%.
- Логика и обоснование: 20%.
- Корректность терминологии и формулировок: 10%.

Правила:
- Каждый критерий оценивается отдельно.
- Пользователь получает детализацию по критериям.
- Помимо балла система обязана выдать 2-4 рекомендации по улучшению.

## 10. Измеримые KPI/SLO MVP

### 10.1. KPI качества RAG
- `KPI-R1` Доля ответов с источниками: **>= 95%** от всех неотказных ответов.
- `KPI-R2` Доля корректных отказов на контрольном наборе: **>= 85%**.
- `KPI-R3` Доля фактических ошибок (галлюцинаций) на пилотном gold-set: **<= 10%**.

### 10.2. KPI качества self-check
- `KPI-S1` Средняя абсолютная ошибка автооценки открытых ответов против экспертной: **<= 1.0** балла по шкале 0..5.
- `KPI-S2` Доля заданий с осмысленной обратной связью (не пустой шаблон): **>= 95%**.

### 10.3. SLO эксплуатации
- `SLO-O1` p95 latency для Q&A ответа: **<= 15 сек**.
- `SLO-O2` Доля запросов с технической ошибкой (5xx/timeout): **<= 3%**.
- `SLO-O3` Успешность ingestion-job (без критических ошибок): **>= 95%**.

### 10.4. KPI агентной оркестрации
- `KPI-A1` Точность маршрутизации `Planner` на размеченном eval-наборе: **>= 90%**.
- `KPI-A2` Доля корректных блокировок `Verifier` (hard-gate) на low-context/off-topic наборе: **>= 85%**.
- `KPI-A3` Доля self-check попыток с валидным payload оценки (без `invalid_evaluation`): **>= 95%**.

## 11. Критерии приемки MVP

MVP принимается, если одновременно выполнено:
- Реализованы роли и RBAC-проверки в Web/Telegram/API.
- Администратор может загрузить пакет файлов поддерживаемых форматов.
- После индексации пользователь получает ответ с корректными источниками.
- При нерелевантном контексте система отказывает по политике раздела 8.
- Self-check формирует оба типа вопросов и оценивает открытые ответы по рубрике раздела 9.
- В `Q&A` используется маршрут `Planner -> Retrieval -> Answer -> Verifier`.
- Для отказов и технических сбоев логируются причины (`reason_code`).
- Система запускается локально по инструкции.
- Основной демонстрационный сценарий проходит через web-интерфейс.
- Достигнуты минимальные KPI/SLO из раздела 10.

## 12. Риски и меры снижения

- Риск потери формул при извлечении текста из PDF.
  - Мера: отдельный quality-check для документов с формулами, fallback на ручную разметку важных разделов.
- Риск нестабильности LLM API.
  - Мера: таймауты, retry, шаблон деградации с техническим сообщением пользователю.
- Риск ошибочной автопроверки открытых ответов.
  - Мера: регулярная калибровка на экспертно размеченной выборке.

## 13. Этапы реализации

1. Схема данных, RBAC и базовый API-контур.
2. Ingestion-pipeline и pgvector индекс.
3. Q&A с guardrails и источниками.
4. Self-check и рубрика оценивания.
5. Web-рендер формул и доработка UX.
6. Замеры KPI/SLO, стабилизация, приемка.
