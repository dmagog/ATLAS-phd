# ATLAS phd: Design Roadmap (UI/UX до уровня защиты)

> **Версия:** 0.1 (драфт)
> **Статус:** живой документ, прорабатывается итеративно
> **Контекст:** функциональный backend и базовый Jinja2-UI отгружены (release v0.7.0 «pilot-ready»). Этот план описывает превращение продукта в **demo-grade артефакт для повторной защиты**, где комиссия не имеет возможности тестировать систему вживую и опирается на визуальный материал и устный рассказ.
> **Позиционирование цели:** не «удобный продукт для аспирантов» (это побочный эффект), а **визуально упакованная демонстрация диссертационного результата**. Каждое UI-решение оценивается через два критерия: (1) как это выглядит на скриншоте/слайде; (2) как это поддерживает нарратив диссертации.

---

## 1. Контекст и зафиксированные развилки

### 1.1 Точка отсчёта

- 6+ Jinja2-шаблонов, минимальный CSS, одна логотипная картинка в `static/`.
- Отдельная иконография в [`docs/branding/`](branding/) (svg + варианты), включая палитру «calm-blue» и форму shield.
- Multi-tenant с 4 ролями: super-admin / tenant-admin / supervisor / student.
- Q&A с hard-gate retrieval refusal'ом (M3.A.0), self-check с рубрикой 40/30/20/10, eval-harness с измеренными показателями (`refusal_tnr 1.000 vs 0.000`, `κ_binarized = 1.000`).

### 1.2 Зафиксированные развилки (lock на 2026-05-07)

| # | Развилка | Решение | Обоснование |
|---|---|---|---|
| 1 | Стек | Jinja2 + HTMX + Tailwind | Соответствие M1/8 GB ограничению; нулевой build; быстрая итерация без рассинхрона дизайн/код |
| 2 | Источник дизайна | Design-in-code | Нет отдельного дизайнера и брендбука; solo + LLM-ассистент быстрее без Figma-промежуточного слоя |
| 3 | Бренд | Развитие существующего ATLAS-shield + «calm-blue» | В `docs/branding/` уже наработана осмысленная иконография; ребрендинг не нужен |
| 4 | Глубина юзабилити | Best practices без живых тестов | Тонкая донастройка — после защиты, в эксплуатации |
| 5 | Целевая аудитория | Не релевантна | Защита по визуалу/рассказу; реальные пользователи появятся после пилота |

### 1.3 Принципы дизайн-решений

1. **Screenshot-first.** Каждый ключевой экран проектируется под идеальный кадр 16:9 / 16:10 (1440–1920px). Никаких длинных таблиц, обрезанных модалок, пустых hero. Mobile viewport — out-of-scope этого цикла.
2. **Нарратив диссертации → UI-якоря.** Ключевые цифры из M3 должны иметь выделенный визуальный носитель **внутри продукта** (не только в README/слайдах):
   - дашборд `/eval` с baseline vs verifier и per-topic breakdown;
   - badge «Hard-gate активен» в Q&A с раскрытием при hover;
   - визуализация рубрики 40/30/20/10 в self-check с конкретными баллами по сессии.
3. **Refusal — first-class экран.** Это не error state, а демонстрация ключевого результата диссертации (защита от галлюцинаций). Проектируется как полноценный экран с объяснением, что произошло и почему это правильное поведение.
4. **Dark mode как default.** На проекторе и в записи демо тёмная тема почти всегда выигрышнее. Light mode — поддерживается, но не приоритет полировки.
5. **Demo data как артефакт.** Sidесcript должен давать «отполированный» датасет: осмысленные имена, реалистичный progress, аккуратные названия материалов, заранее подобранные вопросы с гарантированно красивыми ответами + 1–2 запрограммированных refusal-кейса.

## 2. Фазы

### Фаза 1 — Audit и benchmark (1 день)

**Цель:** зафиксировать стартовую точку и demo-script, который определит приоритеты дальнейших фаз.

**Scope:**
- Скриншот-аудит всех существующих экранов под каждой из 4 ролей. Маркеры: «сломано», «неочевидно», «отсутствует».
- Карты пользовательских путей (user journeys) на роль: что такое «успешная сессия» для студента / супервизора / админа.
- Бенчмаркинг 3–4 аналогов под критерий «как круто выглядит на демо»: Linear, Vercel dashboard, NotebookLM, Perplexity Spaces (не «что удобно», а «что в кадре»).
- **Demo-script outline** — 5–7 шагов будущей защиты с привязкой к экранам. Это превращается в приоритизацию P0 / P1 / P2.

**Артефакты:**
- [`docs/design/ux-audit.md`](design/ux-audit.md) — скриншоты + дефекты + рекомендации по экранам.
- [`docs/design/competitive-scan.md`](design/competitive-scan.md) — паттерны из аналогов.
- [`docs/design/demo-script.md`](design/demo-script.md) — пошаговый сценарий защиты + список экранов P0/P1/P2.

**Definition of done:** demo-script прочитывается как связный нарратив диссертации; список P0-экранов согласован.

---

### Фаза 2 — Информационная архитектура и дизайн-система (1–2 дня)

**Цель:** зафиксировать визуальный язык и набор переиспользуемых компонентов, чтобы дальше не править токены поэкранно.

**Scope:**
- **IA**: sitemap по ролям, навигация (sidebar / topbar), модель пустых состояний, модель ошибок (включая first-class refusal).
- **Дизайн-токены**: цвета (light + dark), типографика (один шрифт UI + моноширинный для цитат/формул), spacing scale, радиусы, тени, motion. CSS-переменные в едином `static/atlas.css`.
- **Базовые компоненты** (~12–15 штук, закрывают 90%): button, input, card, badge, table, modal, toast, breadcrumb, tabs, citation-chip, evidence-block, role-banner, hero-card (для презентации цифр), data-viz-bar (для baseline-vs-verifier).
- Tailwind config + интеграция через CDN или локальный standalone-CLI (без Node/npm в основном образе).
- Демо-страница `/_/styleguide` — доступна только super-admin, показывает все компоненты во всех состояниях. Это и storybook, и regression-тест.

**Артефакты:**
- [`docs/design/design-system.md`](design/design-system.md) — описание токенов, компонентов, ролей.
- `src/atlas/static/atlas.css` + `src/atlas/static/atlas.tailwind.css`.
- `src/atlas/templates/_styleguide.html` + роут `/_/styleguide`.

**Definition of done:** styleguide-страница рендерится; все токены вынесены в CSS-переменные; Tailwind работает через выбранный механизм.

---

### Фаза 3 — Wireframes (low-fidelity) ключевых экранов (1 день)

**Цель:** зафиксировать структуру каждого P0-экрана до визуального полирования.

**Scope:**
- Wireframes делаются прямо в Jinja2 — серые блоки, без визуала, с грубой версткой и плейсхолдер-данными. Это позволяет сразу проверить гипотезу в браузере.
- Порядок — по demo-script из Фазы 1, не по частоте использования.
- Минимальный список:
  1. Landing / auth.
  2. Q&A: вопрос → ответ с inline-цитатами → панель источников → история сессии.
  3. Refusal-экран как отдельный first-class screen.
  4. Self-check: генерация → прохождение → разбор по рубрике с визуализацией 40/30/20/10.
  5. `/eval` дашборд: baseline vs verifier, per-topic breakdown.
  6. Supervisor dashboard: список студентов → progress по топикам → drill-down.
  7. Materials (admin): загрузка → прогресс ingestion → список с состоянием.
  8. Onboarding / invite-flow (M4.A).

**Артефакты:** ветка `feature/design-wireframes`, скриншоты в [`docs/design/wireframes.md`](design/wireframes.md).

**Definition of done:** структура каждого P0-экрана прокликивается в браузере и согласована.

---

### Фаза 4 — Визуальный дизайн (high-fidelity) (2–3 дня)

**Цель:** превратить wireframes в визуальный продукт защитного уровня. Самая важная и ресурсоёмкая фаза.

**Scope:**
- Применение дизайн-системы к каждому wireframe. Dark mode — обязателен и приоритетен.
- Иконография: lucide или phosphor (по итогам выбора в Фазе 2). Бренд-иконка из [`docs/branding/`](branding/) — в favicon, auth-страницы, splash.
- Микро-анимации только функциональные: loading retrieval, появление цитат при hover, плавный переход в refusal-экран.
- Data-viz: bar-chart для baseline-vs-verifier (без heavy chart-библиотек — собираем на чистом SVG/CSS, чтобы не утяжелять стек).
- Hero-cards для презентации цифр (`refusal_tnr 1.000`, `κ_binarized 1.000`, `<2s hard-gate`) — не просто числа, а визуальная композиция с подписью и сравнением.

**Артефакты:** все P0-экраны имплементированы в визуальном стиле, скриншоты обновлены в [`docs/design/wireframes.md`](design/wireframes.md) → переименовано в `docs/design/screens.md`.

**Definition of done:** P0-экраны выглядят как продукт, а не как PoC; скриншоты пригодны для слайдов без дополнительной обработки.

---

### Фаза 5 — Реализация по экранам (5–8 дней)

**Цель:** довести до production-качества все экраны (P0 + P1), оставив P2 на функциональном уровне.

**Порядок (по demo-impact, не по частоте использования):**

| # | Экран | Приоритет | Заметки |
|---|---|---|---|
| 1 | `base.html` + nav + auth | P0 | Первый кадр; задаёт визуальный язык |
| 2 | Q&A с citation-panel + hard-gate badge | P0 | Лицо продукта |
| 3 | Refusal-экран | P0 | First-class, аргумент диссертации |
| 4 | Self-check с визуализацией рубрики | P0 | Демонстрирует κ-результат |
| 5 | `/eval` дашборд (baseline vs verifier, per-topic) | P0 | Носитель центральной цифры M3 |
| 6 | Supervisor dashboard | P1 | Демонстрирует ценность multi-tenant + ролей |
| 7 | Materials / admin / ingestion progress | P1 | Демонстрирует RAG-pipeline |
| 8 | Styleguide / settings / профиль | P2 | Функциональны, без излишней полировки |

**Каждый экран:**
- Отдельный коммит + push (правило из CLAUDE.md).
- Accessibility-чек: keyboard nav, aria-labels, контраст AA минимум.
- Проверка во всех ролях, где экран доступен.

**Definition of done:** P0 + P1 экраны имплементированы; styleguide синхронизирован с реальными экранами.

---

### Фаза 6 — Demo packaging (2–3 дня)

**Цель:** упаковать готовый продукт в защитные артефакты. **Замена живого юзабилити-тестирования** (которое не нужно по контексту защиты).

**Scope:**

1. **Куратированный seed для демо.**
   - `scripts/seed_demo.sh` — заполняет систему «отполированным» датасетом: осмысленные имена студентов, реалистичный progress по 6 топикам, аккуратные названия материалов из текущего корпуса (Born&Wolf / Матвеев / Yariv).
   - Заранее подобранные вопрос-ответ-сессии (Q&A) с красивыми цитатами.
   - 2 demo-вопроса в self-check с разными уровнями оценок (показать рубрику в работе).
   - 2 demo refusal-кейса (off-topic вопрос → hard-gate сработал).

2. **Скриншот-галерея для слайдов.**
   - [`docs/design/screenshot-gallery.md`](design/screenshot-gallery.md) — аннотированный набор скриншотов каждого P0-экрана с подписями «что показывает» и «какой тезис диссертации поддерживает».
   - Опционально: 1–2 GIF/screencast 10–15 секунд для самых динамичных моментов (retrieval → ответ с появлением цитат, refusal-detection).

3. **Design rationale для комиссии.**
   - [`docs/design/rationale.md`](design/rationale.md) — короткое (1–2 страницы) обоснование UX-решений в академических терминах: Nielsen heuristics, WCAG AA, информационная плотность, разделение ролей. На случай вопросов комиссии.

4. **Финальный pass.**
   - Пустые состояния (0 материалов, 0 сессий, 0 студентов).
   - Длинные имена / длинные вопросы / длинные ответы.
   - Все три «edge» состояния retrieval: 0 evidence (refusal), 1 чанк (warning), всё ок.
   - Сетевые ошибки (LLM down, embeddings down).

**Definition of done:** seed работает с одной команды; галерея покрывает все P0-экраны; rationale пригоден к печати.

## 4. Риски

| # | Риск | Митигация |
|---|---|---|
| R1 | Tailwind через CDN ломается на проекторе без интернета | Локальный standalone-CLI генерирует production-CSS, кладём в `static/`; CDN — только на dev |
| R2 | Скриншоты выходят слишком «PoC» из-за плотных таблиц | Phase 4 в первую очередь работает над hero-cards и data-viz, не над таблицами |
| R3 | Demo-data выглядит фейково (random имена, нелогичный progress) | Seed-скрипт пишется как куратированный fixture, а не генератор |
| R4 | Hard-gate refusal в реальном демо срабатывает на «правильный» вопрос | Demo-script содержит заранее подобранные кейсы; refusal-кейсы помечены явно |
| R5 | Dark mode на проекторе теряет контраст | Phase 4 проверяет на ноутбуке + внешнем дисплее с пониженной яркостью |
| R6 | Реализация P1-экранов съедает время P0-полировки | Жёсткий приоритет: P0 экраны должны быть production-grade до начала работы над P1 |
