# ATLAS phd: BDD-сценарии

> **Версия:** 0.6 (драфт)
> **Назначение:** behavioral spec, по которому проверяется соответствие реализации задаче. Каждый сценарий должен быть автоматизируемым (или иметь явный manual-вариант проверки) и трассируемым на milestone из [roadmap.md](roadmap.md).
> **Не заменяет:** [use_cases.md](use_cases.md) (concept-level) и [acceptance_tests.md](acceptance_tests.md) (test plan). BDD — операционный слой между ними.

> **Позиционирование.** ATLAS — **обучающая среда**, а не аттестационный сервис. Self-check — инструмент самопроверки и learning feedback, а не финальная оценка. Heatmap у научрука — карта тем, где аспирантам нужна помощь, а не reporting о «провалах». Поэтому в сценариях нет апелляций, формальных оценочных решений, аттестационных комиссий и сертификатов. Реальный кандэкзамен сдаётся за пределами системы; ATLAS только помогает к нему готовиться.

## 0. Соглашения

### Язык
Gherkin с русскими ключевыми словами: `Функция / Сценарий / Контекст / Дано / Когда / Тогда / И / Но`.

### Теги
- `@m3 @m4 @m4.5 @m5 @m6` — milestone, на котором сценарий становится тестируемым.
- `@negative` — негативный или edge-сценарий.
- `@privacy` — затрагивает privacy-политику.
- `@nfr` — non-functional requirement (latency, isolation, throughput).
- `@open-question:N` — реализация заблокирована open question N из [roadmap.md §4](roadmap.md).

### Акторы (см. [roadmap.md §1.4](roadmap.md))
- **super-admin** — владелец платформы; создаёт тенанты.
- **tenant-admin** (методист кафедры) — управляет корпусом и программой одного направления.
- **supervisor** (научрук) — видит аналитику группы.
- **аспирант** — основной пользователь.

### Базовые сущности
- `tenant` = направление кандэкзамена (`optics-kafedra`, `physics-kafedra`, ...).
- `program` — структурированный документ с разделами и темами/билетами.
- `topic` (билет) — единица программы; материалы привязываются к темам.
- `material` — загруженный источник (учебник/конспект); содержит chunks с topic-разметкой.
- `session` — Q&A-диалог пользователя.
- `attempt` — попытка self-check.

## 1. Функция: Q&A в рамках направления

> Эволюция UC-01. Добавляются tenant-скоуп, topic-фильтр, диалоговая непрерывность.

```gherkin
Контекст:
  Дано существует тенант "optics-kafedra" с загруженной программой и корпусом
  И аспирант "Иван" зарегистрирован в этом тенанте
  И аспирант авторизован
```

### Сценарий 1.1 — Ответ с цитатами по корпусу тенанта `@m4`

```gherkin
Когда Иван задаёт вопрос "Что такое дифракция Френеля?"
Тогда система возвращает ответ за p95 ≤ 15 секунд
И ответ содержит ≥ 1 цитату
И каждая цитата ссылается на material из tenant "optics-kafedra"
И в логах зафиксирован Planner-маршрут "qa" и статус "RESPONSE_SENT"
```

### Сценарий 1.2 — Q&A в режиме билета (с расширительным fallback) `@m4.5`

> Решение: режим билета — отдельный mode с жёсткой фильтрацией retrieval. При недостатке покрытия по билету — асимметричный fallback: для Q&A расширяем поиск, для self-check (2.1) блокируем запуск.

```gherkin
Дано в программе тенанта существует билет "Дифракция"
И coverage билета "Дифракция" ≥ K chunks
Когда Иван задаёт вопрос в режиме билета "Дифракция": "Объясни принцип Гюйгенса-Френеля"
Тогда retrieval использует только chunks, привязанные к билету "Дифракция"
И ответ включает цитаты только из этих chunks
И UI отображает индикатор "режим билета: Дифракция"

# Fallback при тонком покрытии:
Когда coverage билета < K chunks
И Иван задаёт вопрос в режиме билета
Тогда retrieval расширяется на весь корпус тенанта
И UI показывает явную плашку «по этому билету материалов мало, искал шире»
И в логах фиксируется `topic_mode_fallback=true`
```

### Сценарий 1.3 — Отказ при недостатке evidence `@m3`

```gherkin
Когда Иван задаёт вопрос "Какая столица Зимбабве?"
Тогда система возвращает отказ со статусом "REFUSAL_SENT"
И reason_code равен "low_relevance_top1" или "insufficient_evidence"
И система предлагает 1–3 переформулировки
И ответ НЕ содержит фактических утверждений по существу вопроса
```

### Сценарий 1.4 — Изоляция тенантов на retrieval `@m4 @nfr @negative`

```gherkin
Дано существует тенант "physics-kafedra" с непересекающимся корпусом
Когда Иван (тенант "optics-kafedra") задаёт любые 10 вопросов
Тогда ни один retrieval candidate не относится к тенанту "physics-kafedra"
И при попытке прямого API-доступа к material другого тенанта возвращается 403/404
```

### Сценарий 1.5 — Уточняющий вопрос в той же сессии `@m4`

```gherkin
Дано Иван задал вопрос и получил ответ в session_id = S1
Когда Иван в той же сессии задаёт уточнение "А в чём отличие от дифракции Фраунгофера?"
Тогда система использует контекст предыдущего обмена при retrieval
И ответ согласуется с предыдущим (не противоречит ранее высказанным фактам)
```

### Сценарий 1.6 — Корректный рендер формулы `@m4`
> Эволюция UC-04.

```gherkin
Когда Иван задаёт вопрос "Запиши уравнение эйконала"
Тогда ответ содержит формулу в LaTeX
И в web UI формула рендерится как изображение/MathML
И формула не теряется при копировании текста ответа
```

### Сценарий 1.7 — Недоступность LLM-провайдера `@m4 @negative @nfr`

```gherkin
Дано LLM API недоступен
Когда Иван задаёт вопрос
Тогда система возвращает controlled error со статусом "TECHNICAL_ERROR"
И сообщение содержит request_id для саппорта
И ошибка зафиксирована в structured-logs с уровнем ERROR
И UI показывает понятное «временно недоступно, попробуйте через минуту»
```

### Сценарий 1.8 — Сигнал о плохом ответе от пользователя `@m4`

```gherkin
Дано Иван получил ответ на вопрос
Когда Иван нажимает «этот ответ некорректен» и опционально оставляет комментарий
Тогда сигнал записан в таблицу `user_feedback` с request_id, user_id, timestamp, comment
И сигнал виден в админ-панели tenant-admin'а соответствующего тенанта как кандидат на добавление в eval-set
И UI подтверждает «спасибо, мы посмотрим»
```

### Сценарий 1.9 — Раскрытие исходника по цитате `@m4`

```gherkin
Дано Иван получил ответ с цитатой на material "born-wolf.pdf", страницу 124
Когда Иван нажимает на цитату
Тогда открывается читаемый фрагмент исходника (страница ± 1) с подсветкой релевантного chunk'а
И фрагмент содержит метаданные material'а (название, автор, страница)
И при отсутствии прав на полнотекстовый просмотр (например, для chunks с copyright-restricted) видна минимум сама цитата + ссылка на bibliographic запись
```

## 2. Функция: Self-check по программе

> Эволюция UC-02. Self-check теперь привязан к темам программы, а не к свободной строке.

```gherkin
Контекст:
  Дано тенант "optics-kafedra" с программой
  И аспирант авторизован
```

### Сценарий 2.1 — Self-check по выбранному билету `@m4.5`

> В отличие от Q&A в режиме билета (1.2), self-check НЕ имеет fallback на расширение поиска: если материалов мало, не запускается вовсе.

```gherkin
Дано coverage билета "Дифракция" ≥ K chunks
Когда Иван запускает self-check по билету "Дифракция"
Тогда система генерирует ≥ 3 multiple-choice и ≥ 2 open-ended вопроса
И все вопросы grounded в материалах билета "Дифракция" (`grounded=true` в логах)
И UI показывает прогресс по вопросам

# Блокировка при тонком покрытии:
Когда coverage билета < K chunks
И Иван открывает страницу билета (3.5)
Тогда кнопка «запустить self-check» дизейблена
И tooltip объясняет: «для self-check требуется ≥ K материалов по теме; обратитесь к tenant-admin»
```

### Сценарий 2.2 — Возврат оценки с разбором `@m4.5`

```gherkin
Дано Иван завершил self-check (5 вопросов)
Когда Evaluator обработал ответы
Тогда возвращается overall_score 0..5
И возвращаются criterion_scores по 4 критериям (correctness/completeness/logic/terminology)
И для каждого вопроса есть status и комментарий
И evaluator_summary содержит 2–4 предложения
И attempt сохранён в истории Ивана
```

### Сценарий 2.3 — Невалидная оценка `@m4.5 @negative`

```gherkin
Дано Evaluator вернул payload, не проходящий schema-валидацию
Когда система обрабатывает ответ Evaluator
Тогда attempt маркируется "invalid_evaluation"
И финальный результат пользователю НЕ публикуется
И возвращается controlled error с предложением перезапустить попытку
И событие зафиксировано в logs качества
```

### Сценарий 2.4 — Разбор сразу после self-check `@m4.5`

> Ядро learning-цикла. Разбор показывается **по умолчанию вместе с оценкой**, не «по запросу». Без этого self-check = оценка без обучения.

```gherkin
Дано Иван завершил self-check и получил оценку
Когда экран оценки отображается
Тогда вместе с overall_score сразу показан разбор всех вопросов одной страницей:
  - per-question status (зачтён/не зачтён) и его ответ
  - для каждого вопроса — кнопка «подробнее»
Когда Иван нажимает «подробнее» по конкретному вопросу
Тогда раскрывается:
  - правильный ответ (для multiple-choice — ключ; для open — эталонные критерии)
  - цитата из материала с ссылкой на страницу (при клике работает сценарий 1.9)
  - короткое объяснение, почему ответ зачтён/не зачтён
И эти данные одинаково доступны для верных и неверных ответов
```

### Сценарий 2.5 — Возобновление self-check после прерывания `@m4.5`

```gherkin
Дано Иван начал self-check (5 вопросов), ответил на 2 и закрыл вкладку
Когда Иван возвращается в систему в течение 24 часов
Тогда система предлагает «продолжить незавершённую попытку» с указанием темы и прогресса
И при выборе «продолжить» Иван попадает на 3-й вопрос с сохранёнными ответами 1–2
И при выборе «начать заново» прогресс обнуляется, незавершённый attempt помечен «abandoned»
```

## 3. Функция: Личная подготовка аспиранта

> Минимальная версия §7.5 product-proposal: история и статусы билетов IN-scope; адаптивная стратегия — out.

### Сценарий 3.1 — Первый вход аспиранта (first-time experience) `@m4.5`

```gherkin
Дано Иван только что зарегистрировался в тенанте, attempts = 0
Когда Иван открывает дашборд впервые
Тогда система показывает empty state с:
  - кратким (1–2 предложения) объяснением, что можно делать
  - предложением «начать с билета №1» как первое действие
  - ссылкой на полный список билетов программы
И не показывается пустой heatmap или статистики, у которых нет смысла на нулевых данных
```

### Сценарий 3.2 — Дашборд готовности по программе `@m4.5`

```gherkin
Дано Иван прошёл self-check по 5 из 25 билетов программы
Когда Иван открывает дашборд "Моя подготовка"
Тогда отображается список всех 25 билетов
И для каждого пройденного билета показан последний overall_score и дата
И билеты без попыток помечены как "не пройдено"
```

### Сценарий 3.3 — Возврат к билету `@m4.5`

```gherkin
Дано Иван проходил билет "Интерференция", получил score = 2
Когда Иван выбирает "пройти повторно" из дашборда
Тогда генерируется новый attempt по тому же билету
И в истории сохраняются оба attempt отдельно
```

### Сценарий 3.4 — Управление видимостью для научрука `@m5 @privacy`

```gherkin
Дано Иван имеет статус видимости "anonymous-aggregate-only" (default)
Когда Иван переключает opt-in на "show-to-supervisor"
Тогда персональный прогресс Ивана становится виден его supervisor
И событие записано в audit log с timestamp
И Иван может в любой момент вернуть статус обратно
И при возврате прежний доступ научрука прекращается немедленно
```

### Сценарий 3.5 — Страница билета как точка входа в подготовку `@m4.5`

> Билет — основная единица потребления. Эта страница склеивает Q&A и self-check вокруг одной темы.

```gherkin
Дано Иван открыл билет "Дифракция" из программы или дашборда
Когда страница билета загружается
Тогда отображается:
  - название билета и его положение в программе
  - краткий список ключевых концептов из программы
  - количество и список привязанных к билету материалов (с возможностью раскрыть фрагмент через сценарий 1.9)
  - его собственная история attempts по этому билету (последний score, дата)
И на странице доступны два action'а:
  - «Задать вопрос» (открывает Q&A в режиме билета — сценарий 1.2)
  - «Запустить self-check» (сценарий 2.1)
```

## 4. Функция: Онбординг направления

> Был UC-03 (загрузка). Теперь — серия сценариев для подготовки tenant'а к пилоту.

### Сценарий 4.1 — Создание тенанта super-admin'ом `@m4`

```gherkin
Дано super-admin авторизован
Когда super-admin создаёт тенант со slug "optics-kafedra" и display_name "Кафедра оптики"
Тогда тенант создан в БД
И super-admin получает invite-код для tenant-admin
И tenant виден в админ-панели платформы
```

### Сценарий 4.2 — Загрузка структурированной программы `@m4.5`

```gherkin
Дано tenant-admin авторизован в тенанте "optics-kafedra"
Когда tenant-admin загружает program.md с разделами и билетами
Тогда программа парсится в структуру: разделы → билеты → ключевые концепты
И каждый билет получает уникальный id
И программа отображается в UI как древовидный список
```

### Сценарий 4.3 — Загрузка источников с разметкой по билетам `@m4.5`

```gherkin
Дано программа загружена
Когда tenant-admin грузит material с указанием списка билетов, к которым он относится
Тогда material chunks получают topic-теги при ingestion
И coverage по этим билетам пересчитывается автоматически
И повторная загрузка того же файла (по SHA-256) не дублирует chunks
```

### Сценарий 4.4 — Coverage-report по программе `@m4.5`

```gherkin
Дано программа содержит 25 билетов
И загружено N материалов с topic-разметкой
Когда tenant-admin открывает страницу "Покрытие программы"
Тогда для каждого билета показано количество привязанных chunks
И билеты с покрытием < K помечены красным
И есть кнопка экспорта отчёта в markdown
```

### Сценарий 4.5 — Отчёт о готовности направления к пилоту `@m6`

> Решение «запускаем пилот или нет» принимает super-admin на основании отчёта; авто-блокировки нет.

```gherkin
Когда super-admin открывает страницу «Готовность к пилоту» для тенанта
Тогда отчёт показывает:
  - покрытие программы (билет → количество chunks)
  - последние метрики на eval-set v2 (faithfulness, citation accuracy, refusal precision, latency p95)
  - количество invited/registered аспирантов и научруков
  - билеты без покрытия — явным красным списком
И каждый блок содержит ссылку на исходные данные
И страница не блокирует никаких действий — это информационный отчёт
```

### Сценарий 4.6 — Приглашение пользователей в тенант `@m4`

```gherkin
Когда tenant-admin генерирует invite-link с указанием роли (одна из: "student", "supervisor", "tenant-admin")
Тогда ссылка одноразовая, действует 7 дней
И при регистрации по ссылке user получает указанную роль в этом тенанте
И ссылка после использования инвалидируется
И tenant-admin видит список выпущенных и непогашенных invite-links со сроком истечения
```

### Сценарий 4.7 — Загрузка дубликата программы `@m4.5 @negative`

```gherkin
Дано программа уже загружена в тенант
Когда tenant-admin пытается загрузить новую версию программы
Тогда система НЕ заменяет программу автоматически
И запрашивает явное подтверждение с предупреждением о существующих attempts
И при подтверждении старая программа архивируется (не удаляется), новая становится активной
И существующие attempt сохраняют ссылку на ARCHIVED-программу
```

### Сценарий 4.8 — Низкое качество распарсенного материала `@m4.5 @negative`

```gherkin
Дано tenant-admin загрузил PDF "lectures-12.pdf"
Когда ingestion завершён
Тогда система автоматически считает quality-score материала (доля chunks с приемлемой длиной, разумной токенизацией, без artefacts от OCR-failures)
И при quality-score < threshold material помечается флагом "low-quality"
И tenant-admin видит этот флаг в списке материалов с подсказкой о возможных причинах
И low-quality material НЕ исключается автоматически из retrieval, но флаг видим в логах
```

### Сценарий 4.9 — Bootstrapping super-admin при первом запуске платформы `@m4`

```gherkin
Дано платформа разворачивается впервые (БД пуста, нет ни одного user)
Когда сервис стартует с переменными ADMIN_EMAIL и ADMIN_PASSWORD
Тогда создаётся ровно один super-admin с указанными учётными данными
И повторный запуск с теми же ENV не создаёт дубликат и не перезаписывает пароль
И последующая смена пароля возможна только через UI самим super-admin (не через ENV)
```

### Сценарий 4.10 — Регистрация по invite-link с consent на хранение данных `@m4 @privacy`

```gherkin
Дано tenant-admin выпустил invite-link для роли "student"
Когда новый пользователь открывает ссылку
Тогда форма регистрации показывает:
  - поля: email, имя, пароль
  - явный consent-блок: «я согласен на хранение моих учебных данных в рамках тенанта X для целей подготовки»
  - ссылку на политику обработки данных (раздел governance.md)
И регистрация невозможна без явного согласия (чекбокс required)
И факт согласия записан в `user.consent_recorded_at` и в audit log
И при регистрации пользователь сразу попадает в правильный тенант с правильной ролью
```

### Сценарий 4.11 — Первый вход tenant-admin (first-time experience) `@m4`

```gherkin
Дано tenant-admin только что зарегистрировался по invite в новом пустом тенанте
Когда tenant-admin открывает админ-панель тенанта впервые
Тогда отображается empty state с пошаговым чек-листом:
  - [ ] Загрузить программу кандэкзамена
  - [ ] Загрузить ≥ 1 источника
  - [ ] Дождаться, пока coverage по программе достигнет порога K
  - [ ] Пригласить научрука и аспирантов
И каждый пункт — ссылка на соответствующую страницу
И не показываются дашборды, у которых нет смысла на пустых данных
```

### Сценарий 4.12 — Список пользователей тенанта `@m4`

```gherkin
Дано в тенанте есть N зарегистрированных пользователей разных ролей
Когда tenant-admin открывает страницу «Пользователи»
Тогда отображается список с email, ролью, датой регистрации, статусом видимости (для аспирантов), датой последнего входа
И tenant-admin может для каждого пользователя:
  - деактивировать аккаунт (user не сможет залогиниться, данные сохранены)
  - сбросить роль (только если у user'а есть резервный admin/supervisor)
И все эти действия записываются в audit log
И tenant-admin не видит пользователей других тенантов
```

### Сценарий 4.13 — Удаление и обновление material `@m4.5`

```gherkin
Когда tenant-admin удаляет material "lectures-old.pdf"
Тогда chunks этого material исключаются из retrieval немедленно
И existing attempts/sessions сохраняют ссылки на этот material как "deleted-source" (текст ответа не меняется задним числом)
И coverage по затронутым билетам пересчитывается
И факт удаления записан в audit log
Когда tenant-admin загружает новую версию того же material (другой SHA-256)
Тогда старый material архивируется (статус "superseded"), новый становится активным
И existing attempts сохраняют ссылку на superseded-версию
```

## 5. Функция: Аналитика для научрука

### Сценарий 5.1 — Heatmap слабых тем группы `@m5`

```gherkin
Дано в тенанте ≥ 5 аспирантов
И они в сумме сделали ≥ 30 self-check attempts
Когда supervisor открывает дашборд
Тогда отображается heatmap по билетам программы
И билеты упорядочены по убыванию доли неуспешных attempts (score < 3)
И для каждого билета показан confidence interval по доле
И при наведении видно количество attempts (без раскрытия личностей)
```

### Сценарий 5.2 — Drilldown в проблемную тему `@m5 @privacy`

```gherkin
Дано на heatmap отображается билет "Дифракция" с долей неуспешных attempts > 50%
Когда supervisor нажимает на билет "Дифракция"
Тогда открывается детализация по теме:
  - количество attempts всего, успешных, неуспешных
  - частые ошибки по error_tags из self-check evaluator
  - частые формулировки вопросов, на которые система отказывала (refusal_reason)
И персональная информация (кто конкретно не сдал) НЕ отображается, если у студентов нет opt-in
И при N < порога деанонимизации применяются правила сценария 5.3
```

### Сценарий 5.3 — N-порог для агрегатов `@m5 @privacy @negative`

```gherkin
Дано в тенанте < 5 аспирантов
Когда supervisor открывает дашборд
Тогда heatmap НЕ показан
И вместо него — сообщение "недостаточно данных для агрегатов (минимум 5 аспирантов)"
И persona-level данные ни одного аспиранта не отображаются (даже с opt-in)
```

### Сценарий 5.4 — Просмотр персонального прогресса с consent `@m5 @privacy`

```gherkin
Дано аспирант Иван имеет opt-in "show-to-supervisor"
И supervisor авторизован в том же тенанте
Когда supervisor открывает персональную карточку Ивана
Тогда отображается его прогресс по билетам и история attempts
И факт просмотра записан в audit log с supervisor_id, student_id, timestamp
```

### Сценарий 5.5 — Попытка нарушения privacy `@m5 @privacy @negative`

```gherkin
Дано аспирант НЕ давал opt-in
Когда supervisor пытается открыть его персональную карточку (прямым URL)
Тогда возвращается 403
И попытка записана в audit log как "privacy-violation-attempt"
И через UI карточка вообще не доступна (нет ссылки)
```

### Сценарий 5.6 — Первый вход научрука (first-time experience) `@m5`

> Без этого пилотный научрук залогинится первым и увидит «недостаточно данных», что = отток.

```gherkin
Дано supervisor только что зарегистрировался по invite, в тенанте < 5 аспирантов или < 30 attempts
Когда supervisor открывает дашборд впервые
Тогда отображается onboarding-страница с:
  - объяснением, что увидит, когда данные накопятся (превью heatmap на демо-данных)
  - текущим состоянием тенанта: количество аспирантов, attempts, прогноз когда heatmap «оживёт»
  - явной фиксацией privacy-политики: «вы видите агрегаты по группе ≥ 5; персональные данные — только при opt-in аспиранта»
  - ссылкой «попробовать систему как эксперт» (открыть Q&A в режиме supervisor)
И после накопления данных onboarding-страница не показывается, а сразу heatmap (5.1)
```

### Сценарий 5.7 — Список аспирантов тенанта (с opt-in видимостью) `@m5 @privacy`

```gherkin
Дано в тенанте есть аспиранты, часть из которых дала opt-in "show-to-supervisor"
Когда supervisor открывает страницу «Аспиранты»
Тогда отображается:
  - аспиранты с opt-in — с email, датой последней активности, ссылкой на персональную карточку (5.4)
  - аспиранты без opt-in — анонимизированно (например, «Аспирант #3»), без ссылки на карточку, с пометкой «приватный профиль»
И факт открытия списка записан в audit log
И тестовый/деактивированный пользователь не отображается
```

## 6. Функция: Качество и наблюдаемость

> Привязка к M3. Эти сценарии становятся acceptance criteria для eval-harness.

### Сценарий 6.1 — Стартовые таргеты на golden set `@m3 @nfr`

> Floor-метрики, под которые планируется первый прогон. После первого прогона цифры пересматриваются по реальным данным (отдельный sub-doc M3-report.md).

```gherkin
Дано загружен eval/golden_set_v1.0.jsonl (120 элементов)
Когда eval/runner.py отрабатывает в режиме treatment
Тогда faithfulness на Q&A-блоке (60 вопросов) ≥ 0.65
И citation accuracy ≥ 0.80
И refusal precision на refusal-блоке ≥ 0.80
И false refusal rate на Q&A-блоке ≤ 0.20
И self-check rubric MAE ≤ 1.2 по шкале 0..5
```

### Сценарий 6.2 — A/B agentic vs baseline `@m3`

```gherkin
Дано один и тот же golden set
Когда runner отрабатывает в режимах baseline и treatment
Тогда faithfulness(treatment) ≥ faithfulness(baseline)
И delta документирована в M3-report.md
И обе сессии воспроизводимы (фиксированный random seed где применимо)
```

### Сценарий 6.3 — Latency SLA `@m3 @nfr`

```gherkin
Дано серия из 100 Q&A-запросов на golden set
Когда runner собирает latency
Тогда p50 ≤ 8 секунд
И p95 ≤ 15 секунд
И error_rate ≤ 0.03
```

### Сценарий 6.4 — Регрессионный сигнал перед merge `@m3`

> Не hard-gate (judge шумит, false positives заблокируют merge). Warning + ссылка на отчёт; решение принимает разработчик.

```gherkin
Дано baseline-метрики из M3-report зафиксированы
Когда в коде сделано изменение и запущен runner на subset golden set
Тогда CI пишет комментарий в PR с дельтой по основным метрикам
И при просадке faithfulness > 0.05 или latency p95 > 30% PR помечается label "metrics-regression"
И прогон сохраняется в eval/results/{commit_sha}/
```

### Сценарий 6.5 — Воспроизводимость eval-прогонов `@m3 @nfr`

```gherkin
Дано один и тот же golden set, один и тот же commit, один и тот же config
Когда runner отрабатывает дважды подряд в один день
Тогда delta faithfulness между прогонами ≤ 0.03
И delta refusal precision ≤ 0.03
И при превышении этих порогов отчёт фиксирует факт нестабильности judge'а как known limitation
```

## 7. Функция: Governance и privacy

### Сценарий 7.1 — Audit log создаётся для критических действий `@m4 @privacy`

```gherkin
Когда происходит любое из:
  - создание тенанта
  - выдача роли (admin/supervisor)
  - просмотр персональных данных аспиранта
  - изменение consent-статуса
  - удаление material/program
Тогда событие записано в таблицу `audit_log` с actor_id, action, target, timestamp, request_id
И записи в audit_log не редактируемы и не удаляемы через обычные API
```

### Сценарий 7.2 — Экспорт данных тенанта `@m4`

```gherkin
Дано super-admin авторизован
Когда super-admin запрашивает экспорт тенанта "optics-kafedra"
Тогда генерируется архив с: программой, materials (метаданные + content), users (без паролей), attempts, sessions
И архив включает manifest.json со схемой и checksum
И архив не содержит данных других тенантов
```

### Сценарий 7.3 — Удаление данных аспиранта по запросу `@m4 @privacy`

```gherkin
Дано аспирант Иван запросил удаление через UI
Когда tenant-admin подтверждает запрос
Тогда personal data Ивана (email, name, sessions, attempts) удалены
И записи anonymized: attempts остаются с user_id = NULL для агрегатной аналитики
И факт удаления записан в audit log
И аспирант получает email-подтверждение
```

### Сценарий 7.4 — Версионирование программы при изменении `@m4.5`

> Решение: hard archive on replace + UX-разделение архивных attempts от актуальных.

```gherkin
Дано в тенанте есть программа v1 и аспирант с attempts по этой программе
Когда tenant-admin загружает программу v2
Тогда v1 архивируется со статусом "archived"
И существующие attempts остаются привязанными к v1 (без миграции на v2)
И новые self-check используют v2
И в дашборде аспиранта (3.2) attempts по v2 — в основной секции, attempts по v1 — в отдельном блоке «Архив (по предыдущей версии программы)» с явной плашкой
И билеты v1, отсутствующие в v2, в архивных attempts помечены «билет более не в активной программе»
```

### Сценарий 7.5 — Отзыв роли применяется при следующей авторизации `@m4 @privacy @negative`

```gherkin
Дано supervisor имел активную JWT-сессию
Когда tenant-admin отзывает у него роль supervisor
Тогда при следующем запросе с истекшим access-token (или явном logout/login) роль не восстанавливается
И при попытке доступа к supervisor-роутам возвращается 403
И факт отзыва записан в audit log
```

### Сценарий 7.6 — Audit log UI для tenant-admin `@m4 @privacy`

```gherkin
Дано в audit log есть события по разным actor'ам тенанта
Когда tenant-admin открывает страницу «Журнал событий»
Тогда отображается список событий своего тенанта с фильтрами:
  - по actor (user_id, role)
  - по типу действия (создание, удаление, доступ к персональным данным, изменение consent)
  - по диапазону дат
И tenant-admin видит ТОЛЬКО события своего тенанта
И записи невозможно редактировать или удалять через UI
И есть экспорт в CSV
```

## 8. Функция: Платформа — мульти-направления

> Закрепляет §1.4 roadmap. Сейчас отгружаем 1 направление, но архитектура должна быть готова к N.

Когда integration test от лица аспиранта тенанта "optics-kafedra" задаёт 50 разных вопросов
Тогда retrieval НИ РАЗУ не возвращает chunk из тенанта "physics-kafedra"
И тест включён в CI и блокирует merge при провале

  - количество materials и общий объём chunks
  - покрытие программы (% билетов с покрытием ≥ K)
  - статус («идёт onboarding» / «готов» / «активный пилот»)
И для каждого тенанта super-admin может перейти на его страницу готовности (4.5)

## 10. Что НЕ покрывается этим документом (явно)

### Вне scope из-за позиционирования (обучающая среда, не аттестационная)
- Апелляции и пересмотр оценок self-check.
- Формальные аттестационные решения, сертификаты, итоговые экзаменационные баллы.
- Защищённый аттестационный режим (proctoring, anti-cheating).

### Вне scope текущего цикла (зона роста post-M6)
- Cancel/stop генерации ответа в середине.
- Сохранение полезных ответов в избранное / личные заметки.
- End-of-cycle: что с данными аспиранта после сдачи кандэкзамена (архив / удаление / alumni-режим).
- Cost visibility per tenant (потрачено $X на API в этом месяце).
- Сравнение попыток self-check одного аспиранта во времени (learning trends по теме).
- Временной срез / trend на дашборде научрука (как меняется heatmap неделя к неделе).
- Supervisor как пользователь Q&A в режиме эксперта (отдельный mode).
- Cross-feature transitions: автоматическое предложение запустить self-check после Q&A по той же теме.
- Удаление тенанта целиком (сейчас доступен только read-only режим).
- Prod-телеметрия / live-дашборд качества для super-admin (offline через M3 — есть; online — нет).
- История Q&A-сессий аспиранта (возврат к диалогу за прошлый день).
- Поддержка нескольких super-admin'ов на платформу (текущая модель — ровно один super-admin, создаётся при bootstrap).
- **Concept tagging и concept confusions** (бывший сценарий 5.6) — словарь концептов и аналитика «частые путаницы». Решение: пропускаем в M5, M5 строится на per-bilet аналитике. Возвращаемся при наличии запроса от пилотных научруков.

### Вне scope по другим причинам
- Performance под нагрузку > 10 одновременных пользователей.
- Disaster recovery, бэкапы.
- Соответствие конкретным РФ-регламентам персональных данных (отдельный документ governance).
- UX-копирайтинг (тексты ошибок, формулировки кнопок).
- Mobile-флоу.
- UX-структура (главное меню, layout, навигация) — описывается в UX-документации, не в BDD.
- Расширение состояний ошибок 1.7 на каждый зависимый сервис (embeddings, retrieval, evaluator) — общий паттерн «controlled error + request_id» применяется одинаково.
