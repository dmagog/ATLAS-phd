# План приемочных тестов v1.1 (AT)

## 1. Функциональные тесты

### AT-01: Q&A через Telegram
- Проверить, что пользователь получает ответ на валидный вопрос через Telegram.
- Ожидание: ответ возвращается в установленном формате.

### AT-02: Q&A и формулы через web UI
- Задать вопрос с формулой в web UI.
- Ожидание: формула рендерится, ответ содержит источники.

### AT-03: Переключение режимов и профиля ответа
- Переключить `Q&A`/`Self-check` и профиль `кратко/подробно/учебный`.
- Ожидание: режим и профиль применяются к следующему запросу без перезапуска.

### AT-04: Проверка RBAC
- Выполнить админ-операцию под ролью `пользователь`.
- Ожидание: отказ `403/forbidden`.
- Повторить под ролью `администратор`.
- Ожидание: операция доступна.

### AT-05: Валидация форматов загрузки
- Загрузить `pdf/docx/txt/md` и `djvu`.
- Ожидание: поддерживаемые форматы принимаются, `djvu` отклоняется явным сообщением.

### AT-06: Пакетная загрузка и ingestion-job
- Загрузить пакет файлов.
- Ожидание: создан `job_id`, есть статусы по каждому файлу и итоговый статус.

### AT-07: Guardrails блокируют ответ ниже порога
- Подать вопрос с низкой релевантностью.
- Ожидание: содержательный ответ не генерируется.

### AT-08: Полнота источников в ответе
- Подать валидный вопрос с ожидаемым ответом из KB.
- Ожидание: есть файл, section/page, фрагмент/указатель фрагмента.

### AT-09: Структура честного отказа
- Подать вопрос вне корпуса KB.
- Ожидание: отказ содержит причину и 1-3 подсказки переформулировки.

### AT-10: Состав self-check
- Запустить self-check по теме.
- Ожидание: в наборе есть тестовые и открытые вопросы.

### AT-11: Рубрика оценки открытых ответов
- Дать открытый ответ.
- Ожидание: возвращаются общий балл, критерии, комментарии, рекомендации.

### AT-12: RU/EN сценарии
- Прогнать UC-01 и UC-02 на русском и английском.
- Ожидание: корректные ответы на обоих языках.

### AT-13: Формулы в Telegram
- Задать вопрос, требующий формулы, через Telegram.
- Ожидание: формула передается в LaTeX-тексте без потери смысла.

### AT-14: Хранение истории пользователя
- Выполнить Q&A и self-check.
- Ожидание: история и результаты попыток доступны пользователю.

## 2. Нефункциональные тесты

### AT-20: Проверка целевого стека
- Проверить зависимости и конфигурацию сервисов.
- Ожидание: используются Python, aiogram, FastAPI, PostgreSQL, pgvector.

### AT-21: Проверка модульной структуры
- Проверить структуру репозитория.
- Ожидание: модули соответствуют декомпозиции ТЗ.

### AT-22: Smoke-test локального запуска
- Поднять сервисы по инструкции.
- Ожидание: система доступна локально и выполняет базовый сценарий Q&A.

### AT-23: Проверка документации
- Проверить наличие README и API описания.
- Ожидание: документация достаточна для запуска без устных пояснений.

### AT-24: Секреты и безопасность
- Проверить отсутствие секретов в репозитории.
- Проверить RBAC-тесты.
- Ожидание: секреты только в env, RBAC не нарушается.

### AT-25: Наблюдаемость
- Проверить логи и метрики.
- Ожидание: есть request_id, технические метрики и статусы ingestion-job.

### AT-26: Retry/timeout на LLM вызовах
- Смоделировать `429/5xx` и timeout.
- Ожидание: выполняются retry по политике, ошибки возвращаются в controlled form.

### AT-27: Проверка KPI/SLO
- Запустить пилотный тестовый набор.
- Ожидание: достигнуты пороги KPI/SLO из ТЗ.
