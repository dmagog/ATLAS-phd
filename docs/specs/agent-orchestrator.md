# Спецификация агента-оркестратора

## Назначение

`Orchestrator` управляет графом выполнения запроса, фиксирует переходы состояний и гарантирует, что итоговый ответ выходит только через разрешенный путь.

Для PoC оркестратор реализует не свободный multi-agent graph, а маршрутизируемый агентный конвейер:
- `Planner` определяет сценарий;
- после выбора сценария исполняется фиксированная цепочка узлов;
- следующий шаг задается правилами оркестрации, а не автономным решением LLM.

## Шаги PoC

### `Q&A`

1. Принять запрос и создать `request_id`
2. Вызвать `Planner`
3. Вызвать `Retriever`
4. Вызвать `Answer Node`
5. Вызвать `Verifier Node`
6. Завершить сценарий как `RESPONSE_SENT`, `REFUSAL_SENT`, `CLARIFICATION_REQUESTED` или `TECHNICAL_ERROR`

### `Self-check`

1. Принять запрос и создать `request_id`
2. Вызвать `Planner`
3. Вызвать `Self-check Generator`
4. Создать `attempt_id` и сохранить попытку в статусе `created`
5. Зафиксировать состояние `SC_ATTEMPT_CREATED`
6. После генерации questions перейти в `SC_QUESTIONS_READY`
7. После отдельной отправки ответов пользователем валидировать `attempt_id` и payload
8. Перевести попытку в `SC_ANSWERS_SUBMITTED`
9. Вызвать `Self-check Evaluator`
10. Опубликовать валидный результат как `SC_EVALUATED` или завершить `INVALID_EVALUATION`

## Правила переходов

- `Planner` не публикует пользовательский ответ напрямую.
- `Planner` завершает только этап маршрутизации и не управляет дальнейшей логикой шага за шагом.
- `Retriever` не делает fallback в generation без evidence.
- `Verifier` является обязательным шагом для `Q&A`.
- `Evaluator` должен пройти schema validation до публикации результата.
- Оценивание `self-check` разрешено только для существующей незавершенной попытки.
- Повторная отправка уже оцененной попытки должна отклоняться как ошибка идемпотентности / статуса.

## Условия остановки

- успешный ответ с источниками;
- осознанный отказ;
- запрос уточнения;
- контролируемая техническая ошибка.

## Повторные попытки и деградация

- LLM-вызовы: до 2 retry при `429/5xx`
- повторная попытка retrieval по умолчанию не используется; ошибка идет в controlled error
- если planner не уверен, предпочтителен `clarify`, а не агрессивный автозапуск ветки

## Основные входы / выходы

Вход orchestrator:
- `request_id`
- `user_id`
- `session_id`
- `channel`
- `mode_hint`
- `message_text`

Выход orchestrator:
- `final_status`
- `response_payload`
- `reason_code`
- `state_trace`
- `attempt_id` (только для `self-check/start`)
