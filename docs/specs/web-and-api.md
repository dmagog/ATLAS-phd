# Спецификация web-интерфейса и API

## Назначение

`Web UI` и `FastAPI App` образуют основной путь выполнения PoC: пользователь взаимодействует с web-интерфейсом, а backend обрабатывает аутентификацию, маршрутизацию и выдачу результатов.

## Интерфейс

### Единый чат (основной путь)

Главная страница (`/`) — единое текстовое поле. Пользователь пишет вопрос или запрос на самопроверку в свободной форме.

`POST /chat/message` → `Planner` (LLM-классификатор) → один из маршрутов:
- `qa` — запускает `run_qa_flow`, возвращает ответ с цитатами;
- `self_check` — запускает `start_selfcheck`, возвращает inline-квиз прямо в чат;
- `clarify` — возвращает уточняющий вопрос от Planner.

В чате отображается бейдж маршрута (`→ Q&A` / `→ Самопроверка` / `→ Уточнение`).
Квиз отображается inline: пользователь отвечает и жмёт «Отправить ответы» прямо в чате; результаты с разбором появляются там же.

### Отдельные страницы (для прямого доступа и тестирования)

- `/qa` — прямая Q&A без Planner (`POST /qa/message`)
- `/self-check` — прямая самопроверка без Planner (`POST /self-check/start`, `POST /self-check/{id}/submit`)
- `/admin` — управление материалами

## Основные API-сценарии

1. **Единый чат:**
   - пользователь пишет сообщение в свободной форме;
   - `POST /chat/message` → Planner маршрутизирует → ответ, квиз или уточнение.
2. **Прямой Q&A:**
   - пользователь отправляет вопрос;
   - backend возвращает ответ с источниками или отказ.
3. **Self-check start:**
   - пользователь выбирает тему;
   - backend создает попытку и возвращает набор вопросов.
4. **Self-check submit:**
   - пользователь отправляет ответы;
   - backend возвращает оценку и обратную связь.
5. **Admin ingestion:**
   - администратор загружает документы;
   - backend возвращает `job_id` и статусы обработки.

## Поведение аутентификации и RBAC

### Пользовательские маршруты

- требуют валидной пользовательской сессии;
- доступны только собственные `self-check` попытки пользователя.

### Административные маршруты

- требуют роли `admin`;
- дополнительно логируются в аудит;
- без роли `admin` возвращают `403` и не выполняют побочных действий.

## Ожидаемый формат ответов для web-интерфейса

UI должен уметь отрисовывать:
- `answer_markdown`
- `citations[]`
- `LaTeX` формулы
- `refusal` с понятной причиной
- `self-check` результаты с общей и покритериальной оценкой

## Обработка ошибок

Frontend должен различать:
- `refused`
- `error`
- `validation_error`

и обрабатывать их разными UX-сценариями.
