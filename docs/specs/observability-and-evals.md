# Спецификация наблюдаемости и проверок качества

## Цели

PoC должен быть наблюдаемым настолько, чтобы:
- разбирать ошибки процесса выполнения;
- измерять качество `Q&A`, отказов и `self-check`;
- контролировать задержку и стоимость внешнего API.

## Логи

Обязательные поля структурированных логов:
- `timestamp`
- `request_id`
- `user_id` (псевдонимизированный)
- `node_name`
- `state_from`
- `state_to`
- `decision`
- `reason_code`
- `duration_ms`
- `model_id` (если применимо)

Контроли приватности:
- в логи не попадают raw passwords, tokens, secrets;
- пользовательские идентификаторы логируются в псевдонимизированном виде;
- длинные пользовательские ответы и document text не пишутся в стандартные info-level logs целиком;
- для отладочного разбора допускаются только минимальные excerpts, если это нужно для расследования качества.

## Метрики

Минимальный набор:
- `qa_request_count`
- `self_check_request_count`
- `refusal_rate`
- `planner_route_distribution`
- `p95_latency_qa`
- `llm_error_rate`
- `ingestion_job_success_rate`
- `api_token_usage`

## Трассировка выполнения

- Каждый пользовательский запрос имеет `request_id`.
- Внутренние node calls привязываются к этому `request_id`.
- Логи и метрики должны позволять восстановить трассу состояний запроса.
- Для `self-check` дополнительно логируется `attempt_id`.

## Eval-набор

Используется `eval-set v0`, зафиксированный в `docs/product-proposal.md`:
- 120 элементов;
- `Q&A`, refusal, формульно-плотные и self-check кейсы;
- ручная разметка с последующей вторичной проверкой спорных случаев.

## Контроль качества

Проверяем:
- точность маршрутизации;
- долю ответов с источниками;
- долю корректных отказов;
- долю невалидных оценок;
- экспертную полезность обратной связи.

## Разбор отказов

Обязательный ручной разбор для:
- `policy_blocked`
- `invalid_evaluation`
- подозрение на утечку PII
- повторяющиеся промахи retrieval на одной и той же теме
