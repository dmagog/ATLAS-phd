# ATLAS: концепция проекта

## 1. Название и назначение

**ATLAS (Assistant for Technical Learning & Attestation Support)** — интеллектуальный ассистент для подготовки к сдаче кандидатского минимума по технической специальности.

Каналы взаимодействия:
- Telegram-бот (основной канал MVP);
- веб-чат (дополнительный канал MVP, в том числе для отображения формул через LaTeX).

Кандидатский минимум — совокупность обязательных экзаменов в системе подготовки научных и научно-педагогических кадров (аспирантура).

## 2. Проблема

Подготовка к кандидатскому минимуму требует работы с большим объемом разрозненных материалов:
- учебники;
- научные статьи;
- методические пособия;
- конспекты и дополнительные материалы.

Ключевые сложности:
- ручная систематизация информации;
- сложность быстрого поиска релевантных фрагментов;
- отсутствие адаптированной под конкретного обучающегося проверки знаний.

## 3. Целевая аудитория

- аспиранты;
- соискатели ученых степеней;
- слушатели программ подготовки к кандидатским экзаменам.

## 4. Цель проекта

Создать интеллектуального ассистента, который помогает готовиться к кандидатскому минимуму за счет:
- ответов на вопросы по учебным темам;
- самопроверки знаний;
- опоры на ограниченную и управляемую базу учебных материалов.

Пилотная область:
- один конкретный профильный технический экзамен;
- при этом внутри базы могут быть материалы из разных тематических курсов, относящихся к выбранному экзамену.

## 5. Общая концепция решения

ATLAS реализуется как RAG-система (Retrieval-Augmented Generation):
1. Пользователь задает вопрос в Telegram-боте.
2. Система ищет релевантные фрагменты в локальной базе документов.
3. LLM через API формирует связный ответ с опорой на найденный контекст.
4. Ответ сопровождается ссылками на источники (файл, раздел, фрагмент).

Дополнительно система использует LLM для:
- генерации заданий для самопроверки по выбранной теме;
- проверки ответов пользователя и выдачи обратной связи.

## 6. Базовые функциональные возможности (черновой перечень)

### 6.1. Диалоговый интерфейс
- чат-взаимодействие в Telegram и веб-интерфейсе;
- поддержка вопросов по темам подготовки;
- переключаемый режим ответа (кратко/подробно/учебный формат);
- поддержка русского и английского языков;
- базовое сохранение истории Q&A и результатов самопроверки в MVP;
- продвинутая долгосрочная аналитика обучения вынесена в зону расширения.

### 6.2. Роли и доступ
- роль `администратор`: загрузка и обновление базы знаний;
- роль `пользователь`: обучение, ответы на вопросы, самопроверка.

### 6.3. Работа с базой знаний
- загрузка материалов только администратором;
- ручной импорт пакетами;
- допустимые форматы MVP: `pdf`, `docx`, `txt`, `md`;
- формат `djvu` не входит в MVP (может поддерживаться позже через отдельный конвертер/OCR);
- предварительная обработка и индексация;
- векторизация и поиск релевантных фрагментов.

### 6.4. Ответы с опорой на источники
- формирование структурированного ответа;
- подробное указание использованных источников (файл, раздел/глава, фрагмент);
- честный и контролируемый отказ от ответа при недостатке релевантных данных;
- рекомендация переформулировать запрос при отказе.

### 6.5. Самопроверка
- генерация набора вопросов по теме (тестовые + открытые);
- прием ответов пользователя;
- автоматическая проверка, оценка и развивающая обратная связь;
- ограничение по времени в MVP не вводится.

## 7. Нефункциональные требования (черновой перечень)

- язык реализации: Python;
- рекомендуемый стек MVP: `aiogram` + `FastAPI` + `PostgreSQL`;
- модульная архитектура;
- возможность локального запуска;
- документированный код;
- базовые практики безопасности (хранение ключей в переменных окружения, разграничение ролей, аудит действий администратора).

## 8. Базовые пользовательские сценарии

### Сценарий 1: Получение ответа
1. Пользователь задает вопрос по теме.
2. Система ищет релевантные материалы.
3. Система возвращает структурированный ответ с источниками.

### Сценарий 2: Самопроверка
1. Пользователь указывает тему.
2. Система генерирует тест/вопросы.
3. Пользователь отвечает.
4. Система проверяет ответы и дает обратную связь.

### Сценарий 3: Отсутствие данных
1. Пользователь задает вопрос.
2. Релевантные материалы отсутствуют или недостаточны.
3. Система явно сообщает об ограничении и предлагает следующий шаг.

## 9. Границы первой версии (MVP, для уточнения)

В первую версию целесообразно включить:
- Telegram-бот и минимальный веб-чат;
- 2 режима: "вопрос-ответ" и "самопроверка";
- работу с ограниченным набором документов;
- базовый RAG-конвейер;
- ссылки на источники в каждом ответе;
- метрики качества RAG и проверки знаний;
- локальный запуск для демонстрации и тестирования.

Отложить на следующие этапы:
- сложная персонализация траектории обучения;
- режим преподавателя;
- расширенная аналитика прогресса и персонализированные дашборды обучения;
- поддержка `djvu` без предварительной конвертации.
